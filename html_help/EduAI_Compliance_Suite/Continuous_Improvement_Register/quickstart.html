üìä Step 3: Add Workflow Steps

Break down the improvement action into specific, trackable steps with owners and due dates.

Create First Step

POST /api/v1/continuous_improvement/steps/
Authorization: Bearer <your_token>
Content-Type: application/json

{
  "improvement_action_id": 123,
  "sequence_order": 1,
  "title": "Analyze current assessment feedback process",
  "description": "Document current workflow, identify bottlenecks, measure average turnaround time",
  "status": "in_progress",
  "owner_id": 15,
  "due_date": "2025-11-11"
}

Create Second Step

POST /api/v1/continuous_improvement/steps/
Authorization: Bearer <your_token>
Content-Type: application/json

{
  "improvement_action_id": 123,
  "sequence_order": 2,
  "title": "Design improved feedback workflow",
  "description": "Create new process with clear SLA, automated reminders, escalation path",
  "status": "not_started",
  "owner_id": 15,
  "due_date": "2025-11-18"
}

Create Third Step

POST /api/v1/continuous_improvement/steps/
Authorization: Bearer <your_token>
Content-Type: application/json

{
  "improvement_action_id": 123,
  "sequence_order": 3,
  "title": "Implement and train staff on new process",
  "description": "Roll out new workflow, conduct training sessions, update documentation",
  "status": "not_started",
  "owner_id": 28,
  "due_date": "2025-11-25"
}

Step Management Tips

  • Sequence Order: Use increments of 10 (10, 20, 30) to allow inserting steps later
  • Due Dates: Set realistic deadlines considering dependencies and workload
  • Owners: Assign to individuals who have authority and capacity to complete
  • Status Tracking: Update status as work progresses (not_started ‚Üí in_progress ‚Üí completed)
  • Blockers: Use is_blocked flag and blocker_description to track impediments

üí° Best Practices

Action Creation

  • Clear Titles: Use descriptive, actionable titles (e.g., "Improve X" not just "X")
  • Detailed Descriptions: Include context, current state, desired state, and why it matters
  • Source Tracking: Always specify the source (audit, complaint, feedback, review, etc.)
  • Priority Setting: Use priority consistently: Critical = registration risk, High = compliance impact
  • Assignment: Assign to owners who have authority and capacity to act

Step Management

  • Granularity: Steps should be 1-2 weeks of work, not months
  • Dependencies: Order steps logically, considering what must happen first
  • Due Dates: Set realistic deadlines with some buffer for unexpected issues
  • Progress Updates: Use progress_notes field to document work done
  • Blocker Tracking: Flag blockers immediately and document resolution plans

Verification Guidelines

  • Wait Period: Allow 30-90 days after implementation before verifying
  • Evidence Collection: Gather objective data (reports, metrics, surveys)
  • Independence: Verifier should not be the same as implementer when possible
  • Honesty: If not effective, mark as "requires_rework" and create follow-up actions
  • Documentation: Thorough notes help future improvements and audits

‚ùì Troubleshooting

Common Issues

  • AI Classification Not Working: Ensure description has enough detail (at least 50 words recommended). Include keywords related to ASQA standards for better clause linking.
  • SLA Breach Warnings: Check that SLA policies are configured for your action's priority/category. Use GET /api/v1/continuous_improvement/sla-policies/ to review policies.
  • Steps Not Showing: Verify improvement_action_id matches the parent action. Check that sequence_order is set correctly.
  • Verification Fails: Ensure action status is "completed" before creating verification. Verifier must have appropriate permissions.
