Configure AI content generation for TAS documents

What are GPT Prompts?

GPT Prompts are configuration objects that tell the AI system (GPT-4) how to generate content for each section of a TAS document. They define:

  • The AI's role and expertise (system prompt)
  • What content to generate (user prompt template)
  • How creative or factual to be (temperature)
  • How much content to generate (max_tokens)
  • What variables to inject from context
  • How to validate the generated content

üéØ Two Levels of Prompts

You can configure GPT prompts at two levels:

  1. Section-level: Stored in TASTemplateSection.gpt_prompt - applies to this section everywhere
  2. Template-level: Stored in TASTemplate.gpt_prompts[section_code] - overrides section-level for this template

Priority: Template-level overrides section-level. If neither exists, generation fails.

GPT Prompt Structure

A GPT prompt configuration is a JSON object with these fields:

Field Type Required Description
system_prompt String Yes Defines AI's role, expertise, and behavior
user_prompt_template String Yes Instruction with placeholders like {variable_name}
temperature Float No 0.0-1.0, controls creativity (default: 0.7)
max_tokens Integer No Maximum response length (default: 1000)
variables Array No List of variables to inject from context
validation_keywords Array No Keywords that must appear in response

System Prompt Best Practices

Define the AI's Role

Start by establishing who the AI is:

"You are an expert in Australian VET qualifications and ASQA compliance."

Specify Knowledge Areas

Tell the AI what expertise it should draw upon:

"You have deep knowledge of:
- Australian Qualifications Framework (AQF)
- ASQA Standards for RTOs 2015
- Training package development
- Assessment validation procedures"

Set Tone and Style

Guide the AI's writing style:

"You write in clear, professional Australian English suitable for RTO documentation. Your content is concise, well-structured, and compliance-focused."

Industry-Specific System Prompts

Trade/Technical

"You are an expert in Australian trade qualifications with extensive knowledge of workplace-based training, practical assessment, and WHS requirements in vocational education. You understand the hands-on nature of trade training and write accordingly in Australian English."

Business/Management

"You are an expert in business education and Australian VET qualifications, with deep knowledge of management principles, business operations, and professional skills development. You write in professional business language using Australian spelling."

Health/Community Services

"You are an expert in health and community services education with comprehensive knowledge of clinical practice, duty of care, infection control, and person-centered care principles in Australian context. You understand AHPRA requirements and write in clear, caring Australian English."

User Prompt Templates

Variable Injection

Use curly braces to inject variables from context:

"Generate content for {qualification_code} - {qualification_title}"

At generation time, these are replaced with actual values:

"Generate content for BSB50120 - Diploma of Business"

Common Variables

Variable Description Example Value
{qualification_code} Training package code BSB50120
{qualification_title} Full qualification name Diploma of Business
{aqf_level} AQF level diploma
{target_audience} Intended students working professionals, school leavers
{delivery_mode} How delivered blended, workplace-based
{duration} Program length 52 weeks, 18 months
{units} Units of competency BSBWHS311, BSBCMM411, ...
{industry} Industry sector hospitality, construction, healthcare

Structured Instructions

Use numbered lists or bullet points for clarity:

"Generate an Assessment Strategy that includes:
1. Overview of assessment approach
2. Assessment principles (validity, reliability, flexibility, fairness)
3. Rules of evidence (validity, sufficiency, authenticity, currency)
4. Assessment methods to be used
5. Validation and moderation procedures
6. Reasonable adjustments process

Context:
- Qualification: {qualification_code} - {qualification_title}
- AQF Level: {aqf_level}
- Delivery Mode: {delivery_mode}
- Duration: {duration}

Write in professional language suitable for ASQA compliance."

Temperature Guidelines

Temperature controls how creative vs. deterministic the AI's output is:

Range Behavior Best For Examples
0.0 - 0.3 Very focused, deterministic Factual content, compliance Entry requirements, trainer qualifications, ASQA compliance mapping
0.4 - 0.6 Balanced, some variation Structured descriptions Assessment procedures, resource lists, unit sequencing
0.7 - 0.8 Creative, varied Narratives, overviews Qualification overview, industry relevance, career pathways
0.9 - 1.0 Highly creative, unpredictable Brainstorming, examples Sample scenarios, case study ideas (rarely used for TAS)

‚ö†Ô∏è Recommended Range: 0.5 - 0.7

For TAS generation, stay in the 0.5-0.7 range. Too low (<0.4) produces repetitive content. Too high (>0.8) produces inconsistent compliance language.

Max Tokens Guidelines

Max tokens limits response length (1 token ‚âà 0.75 words):

Section Type Typical Word Count Recommended max_tokens
Short descriptions 100-200 words 200-300
Medium sections 300-500 words 500-800
Large sections 600-1000 words 1000-1500
Comprehensive sections 1000-1500 words 1500-2000
Tables/Lists Varies 500-1000

üí° Token Budget

Set max_tokens generously but not wastefully. If section needs 500 words, set to 800-1000 to allow for completeness. GPT will stop naturally when content is complete if temperature is reasonable.

Validation Keywords

Validation keywords ensure generated content covers required topics:

How It Works

  1. AI generates content
  2. System checks if all validation_keywords appear in content (case-insensitive)
  3. If any keyword missing, generation is flagged for review
  4. Human reviewer can approve or regenerate

Examples by Section

Qualification Overview

"validation_keywords": ["purpose", "pathways", "outcomes", "career", "licensing"]

Assessment Strategy

"validation_keywords": ["validity", "reliability", "flexibility", "fairness", "moderation"]

Entry Requirements

"validation_keywords": ["prerequisites", "LLN", "age", "work experience"]

WHS Procedures

"validation_keywords": ["induction", "risk assessment", "PPE", "incident", "emergency"]

‚úÖ Best Practice

Include 3-7 validation keywords that represent must-have topics. Too many keywords make validation too strict. Too few don't provide enough quality control.

Section-Level vs Template-Level Prompts

Section-Level Prompts

Stored in TASTemplateSection.gpt_prompt

‚úÖ Use Section-Level When:

  • Content generation approach is the same across all templates
  • You want consistent section content everywhere
  • The section is truly generic (e.g., "Support Services")
  • You want to update the prompt once for all templates

Template-Level Prompts

Stored in TASTemplate.gpt_prompts[section_code]

‚úÖ Use Template-Level When:

  • Different AQF levels need different language complexity
  • Different industries need different emphasis
  • Template type requires specific compliance mentions
  • You want to override the section's default prompt

Example: Same Section, Different Prompts

Section-Level (Default)

// In TASTemplateSection for ENTRY_REQ
{
  "gpt_prompt": {
    "system_prompt": "You are an expert in VET qualifications...",
    "user_prompt_template": "Generate entry requirements for {qualification_code}...",
    "temperature": 0.5,
    "max_tokens": 500
  }
}

Template-Level Override for Certificate I

// In TASTemplate.gpt_prompts["ENTRY_REQ"]
{
  "system_prompt": "You are an expert in foundation skills education...",
  "user_prompt_template": "Generate simple, plain-language entry requirements for {qualification_code}. Use short sentences. Focus on what students need to know, not formal prerequisites. Be encouraging and supportive in tone.",
  "temperature": 0.6,
  "max_tokens": 300,
  "validation_keywords": ["basic", "simple", "need"]
}

Testing and Iteration

Test Your Prompts

  1. Generate sample content using the prompt
  2. Review for quality: Is it accurate? Complete? Compliant?
  3. Check validation: Do validation keywords appear?
  4. Adjust temperature: Too repetitive? Increase. Too random? Decrease.
  5. Refine instructions: Add more specific guidance if needed
  6. Update max_tokens: Content too short or cut off?

Common Refinements

If Content is Too Generic

// Add more context and examples to user prompt
"Provide specific examples relevant to {industry} sector..."

If Content Lacks Detail

// Increase max_tokens and add more structured requirements
"Include at least 5 specific assessment methods with examples of when each is used..."

If Content is Inconsistent

// Lower temperature and add stricter system prompt
"temperature": 0.5  // was 0.8
"You write in professional, formal language following ASQA documentation standards..."

If Content Misses Key Points

// Add more validation keywords and explicit requirements
"validation_keywords": ["compliance", "ASQA", "standards", "requirements"]
"You must reference ASQA Standards 2015..."

Best Practices Summary

1. Write Clear System Prompts

  • Define role and expertise
  • Specify Australian context and spelling
  • Set tone and style expectations

2. Structure User Prompts Well

  • Use numbered lists for required elements
  • Inject relevant variables
  • Provide context about qualification/template
  • Be specific about what's needed

3. Set Appropriate Temperature

  • 0.5-0.6 for compliance and factual content
  • 0.6-0.7 for descriptive and narrative content
  • Avoid extremes (<0.4 or >0.8)

4. Size Max Tokens Correctly

  • Generous but not wasteful
  • Allow room for completeness
  • Typical range: 500-2000 for TAS sections

5. Use Validation Keywords

  • 3-7 keywords representing must-have topics
  • Mix of compliance and content keywords
  • Test that they actually appear in good content

6. Choose Right Level

  • Section-level for generic, reusable prompts
  • Template-level for specialized or customized prompts
  • Can always override section-level at template-level
