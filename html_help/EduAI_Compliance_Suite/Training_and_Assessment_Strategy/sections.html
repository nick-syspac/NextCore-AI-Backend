What are TAS Sections?

TAS Sections are reusable content blocks that make up a Training and Assessment Strategy document. Think of them as building blocks that can be combined in different ways across multiple templates.

üîÑ Reusability is Key

A single section (like "Entry Requirements") can be used in dozens of templates. This means:

  • Create once, use many times
  • Update the section, and all templates can use the updated version
  • Consistent content across your organization
  • Less duplication, easier maintenance

Section Fields

Field Type Description
section_name String (200) Display name (e.g., "Qualification Overview")
section_code String (50, unique) Unique identifier (e.g., "QUAL_OVERVIEW")
description Text Help text explaining what this section is for
content_type Choice text, rich_text, table, list, json
default_content Text (optional) Example or placeholder content
is_editable Boolean Can users edit after generation?
parent_section FK (optional) For nested/hierarchical sections
gpt_prompt JSONField (optional) AI generation configuration
is_active Boolean Soft delete flag

Content Types

Each section has a content type that determines how content is structured and displayed:

Type Description Best For Example
text Plain text paragraphs Descriptions, overviews, narratives Qualification description, target audience
rich_text Formatted text with HTML Complex content with headings, lists, bold/italic Assessment procedures, detailed instructions
table Structured tabular data Unit mappings, resource lists, schedules Assessment tools table, resource requirements
list Bulleted or numbered lists Requirements, deliverables, options Entry requirements, learning resources
json Structured JSON data Complex data structures, mappings Unit clustering, pathway maps

Creating a Section

API Endpoint

POST /api/tenants/{tenant}/tas/template-sections/

Request Example

{
  "section_name": "Qualification Overview",
  "section_code": "QUAL_OVERVIEW",
  "description": "Provides comprehensive overview of the qualification including purpose, pathways, licensing requirements, and career outcomes",
  "content_type": "rich_text",
  "is_editable": true,
  "default_content": "
Qualification Purpose
[To be generated]
",
  "gpt_prompt": {
    "system_prompt": "You are an expert in Australian VET qualifications and ASQA compliance.",
    "user_prompt_template": "Generate a comprehensive qualification overview for {qualification_code} - {qualification_title}. Include: purpose, licensing requirements, pathways, and career outcomes. Use Australian spelling.",
    "temperature": 0.7,
    "max_tokens": 1000,
    "variables": ["qualification_code", "qualification_title"],
    "validation_keywords": ["purpose", "pathways", "outcomes"]
  }
}

Response Example

{
  "id": 15,
  "tenant": "acme-rto",
  "section_name": "Qualification Overview",
  "section_code": "QUAL_OVERVIEW",
  "description": "Provides comprehensive overview of the qualification...",
  "content_type": "rich_text",
  "default_content": "
Qualification Purpose
[To be generated]
",
  "is_editable": true,
  "parent_section": null,
  "gpt_prompt": { ... },
  "is_active": true,
  "template_count": 0,
  "created_at": "2024-01-20T10:30:00Z",
  "updated_at": "2024-01-20T10:30:00Z"
}

Hierarchical Sections

Sections can have parent-child relationships for nested content:

Example: Assessment Strategy (Parent)

{
  "section_name": "Assessment Strategy",
  "section_code": "ASMT_STRATEGY",
  "content_type": "rich_text",
  "parent_section": null
}

Example: Assessment Principles (Child)

{
  "section_name": "Assessment Principles",
  "section_code": "ASMT_PRINCIPLES",
  "content_type": "rich_text",
  "parent_section": 15  // ID of ASMT_STRATEGY
}

üí° When to Use Hierarchies

Use parent-child sections when:

  • Content logically breaks into subcategories
  • You want to generate sections separately but group in document
  • Different templates need different subcomponents

Duplicating a Section

Create a copy of an existing section as a starting point:

API Endpoint

POST /api/tenants/{tenant}/tas/template-sections/{id}/duplicate/

Request Example

{
  "new_name": "Qualification Overview - Trade Focus",
  "new_code": "QUAL_OVERVIEW_TRADE"
}

This creates an exact copy with a new name and code, including the GPT prompt configuration.

Best Practices

1. Use Clear, Consistent Section Codes

  • Use UPPERCASE_WITH_UNDERSCORES format
  • Keep codes short but meaningful (max 50 characters)
  • Use consistent prefixes (ASMT_ for assessment, DELIVERY_ for delivery)

2. Choose the Right Content Type

  • Text: Simple paragraphs without formatting needs
  • Rich Text: When you need headings, lists, bold/italic
  • Table: Structured data with rows and columns
  • List: Simple bulleted or numbered lists
  • JSON: Complex structured data

3. Write Helpful Descriptions

  • Explain what content should go in this section
  • Include examples or guidelines
  • Reference ASQA clauses if relevant
  • Specify any mandatory inclusions

4. Configure GPT Prompts at Section Level

  • Section-level prompts are more specific and reusable
  • Template-level prompts override section-level when needed
  • Include validation keywords to ensure quality
  • Use appropriate temperature (0.5-0.7 for factual, 0.7-0.9 for creative)

5. Make Sections Truly Reusable

  • Don't hardcode qualification-specific details
  • Use variable placeholders instead
  • Keep sections generic enough for multiple templates
  • Use template-specific settings in templates, not sections
